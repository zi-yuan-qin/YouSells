# 技术选型与部署方案

## 1. 当前定下来的技术栈

### 后端

- `JDK 21`
- `Spring Boot 3.x`
- `Spring Security`
- `MyBatis-Plus`

### 数据层

- `MySQL 8`
- `Redis`

### 前端

- `Vue 3`
- `TypeScript`
- `Vite`
- `Pinia`
- `Vue Router`
- `Element Plus`

### 部署

- `Nginx`
- `Spring Boot 单体应用`
- `Docker Compose`

## 2. 为什么这样选

这套方案适合当前阶段的原因是：

- 技术成熟
- 团队容易接手
- 适合后台管理系统
- 开发效率和维护成本相对平衡
- 对 `50-100` 人内部使用场景完全够用

## 3. 当前不建议做的事

- 一开始就做微服务拆分
- 一开始就做复杂消息系统
- 一开始就把社区功能做得很重
- 一开始就为了“未来扩展”提前引入过多复杂度

## 4. 部署建议

### 方案 A：更稳

- `1 台 Linux 应用服务器`
- `MySQL` 使用托管数据库
- `Redis` 使用托管服务或独立轻量实例
- 应用服务器上跑 `Nginx + Frontend + Backend`

优点：

- 运维压力更小
- 数据库备份恢复更稳
- 更适合当前团队经验

### 方案 B：预算优先

- `1 台 Linux 服务器`
- 使用 `Docker Compose` 跑 `Nginx + Frontend + Backend + MySQL + Redis`

优点：

- 成本更低
- 环境统一
- 起步更直接

风险：

- 数据库和 Redis 需要自己维护
- 备份、恢复、迁移都要自己管

当前更推荐 `方案 A`。

## 5. Redis 当前怎么用

`v1` 不建议重度依赖 Redis，先轻用：

- 缓存热点数据
- 验证码
- 限流
- 临时数据

当前已采用轻量 `JWT Bearer Token` 作为 P0 鉴权基线。Redis 暂不承载登录态，后续如果增强安全性，再用 Redis 接 token 黑名单、刷新状态和多端登录状态。

## 6. 第一版架构原则

- 单体优先
- 前后端分离
- 接口统一
- 数据库正式化
- 有登录鉴权
- 有测试环境和正式环境
- 有基础备份方案

## 7. 后续扩展路径

推荐升级路线：

1. 单体 + 轻量 JWT
2. 单体 + refresh token + Redis 黑名单
3. 网关 + 认证中心 + 业务服务拆分

不要一开始直接按第三步去做。
