# 开发流程与日志规范

## 1. 目标

`YouSells` 项目按正式软件项目流程推进，不按“想到哪写到哪”的方式开发。

每次开发任务都要遵守这条主线：

`明确需求 -> 定开发规划 -> 记录开发日志 -> 开发业务代码 -> 立刻补自动化测试 -> 验证 -> 需要时联调测试`

## 2. 标准开发流程

### 第一步：明确需求

在开始写代码前，先明确：

- 这是哪个模块的任务
- 任务目标是什么
- 属于 `P0 / P1 / P2` 哪一层
- 会不会影响数据库
- 会不会影响接口
- 会不会影响权限
- 会不会影响前端联调

### 第二步：定开发规划

开发前要先定：

- 负责人
- 涉及模块
- 依赖关系
- 验证方式
- 是否需要联调

### 第三步：记录开发日志

每个较完整的任务，都建议留下开发日志。

日志至少包含：

- 日期
- 负责人
- 任务名称
- 当前状态
- 当前进度
- 遇到的问题
- 下一步动作

### 第四步：开发具体业务代码

开发阶段要求：

- 不夹带无关改动
- 不跳过约定流程
- 共享逻辑优先抽离
- 行为变更要及时同步文档

### 第五步：模块完成后立刻写自动化测试

这是强制要求，不是补充要求。

至少要做到：

- 后端模块写单元测试
- 关键服务逻辑写正确性测试
- 核心接口要有接口级验证
- 工具类要有针对性测试

### 第六步：验证

验证至少包括：

- 代码是否能运行
- 测试是否通过
- 功能是否符合需求
- 文档是否同步

### 第七步：联调测试提醒

如果改动影响以下内容，必须提醒开发者做联调测试：

- 页面显示
- 表单提交
- 表格展示
- 登录鉴权
- 权限判断
- 前后端接口联动
- 页面交互行为

## 3. 代码交付要求

一个模块交付时，默认要同时交付：

- 代码
- 自动化测试
- 必要文档更新
- 验证说明

不能只交“功能代码”。

## 4. 角色责任

### 秦梓源

- 负责基础设施准备
- 负责项目工程基线
- 负责阶段性验收
- 负责最终测试验收

### 模块开发者

- 负责自己模块的开发
- 负责自己模块的测试
- 负责自己模块的基本自查

## 5. 当前执行原则

- 先规划，再编码
- 先文档，再实现
- 先模块正确，再追求速度
- 测试必须跟着模块走
- 有前端影响的改动必须提醒联调
