# YouSells 后端架构收口与协作边界规范

日期：`2026 年 5 月 19 日`

---

## 1. 文档目标

这份文档不是推翻当前 `P0` 基线重做架构，而是在正式分发任务前，把 `YouSells` 后端当前最容易失控的几个点先收口：

- 模块边界
- 分层职责
- 跨模块调用规则
- 文件所有权
- P0 阶段允许的临时方案与不允许的写法

目标只有一个：  
让团队在 `P0` 并行开发时，尽量做到 **高内聚、低耦合、少冲突、可验收**。

---

## 2. 当前架构判断

当前 `YouSells` 后端整体方向是正确的：

- 采用按业务域拆分的单体结构
- `common` 承载公共基础能力
- `modules` 按业务模块分包
- `controller -> service -> dto/vo` 基本骨架已存在
- `dashboard` 作为聚合模块独立存在

但当前还不是最终成熟形态，主要问题有：

- 部分模块仍然是样例实现，尚未完全落到真实持久化
- 一些 `service` 仍直接以 `vo` 作为内部处理结构
- 聚合层仍依赖其他模块的分页查询接口做拼装
- 模块目录虽然齐了，但真实边界还需要靠规则约束住

所以这轮收口的定位是：

- 不做大重构
- 不改大方向
- 只把后续多人协作最容易乱掉的边界先定死

---

## 3. 后端总体架构原则

`YouSells` 在 `P0` 阶段采用：

- 前后端分离
- 单体应用优先
- 按业务域分模块
- 公共能力下沉到 `common`
- 聚合能力集中到 `dashboard`
- 真实业务持久化逐模块落地

必须坚持的 5 条原则：

1. 模块按业务边界聚合，不按技术层横向混写。
2. 每个业务模块尽量在自己的目录内闭环完成。
3. 跨模块调用只能依赖对方 `service interface`，不能直接穿透实现类、Controller、Mapper。
4. 聚合需求统一收口到 `dashboard/**`，不把聚合逻辑散落到业务模块。
5. 共享文件默认由队长维护，避免多人同时改高冲突入口。

## 4. 模块边界定义

### 4.1 `common`

职责：

- 配置
- 常量
- 异常
- 统一响应
- 安全
- 公共实体基类

允许放入：

- `config`
- `constant`
- `exception`
- `response`
- `security`
- `entity`

不允许放入：

- 具体业务逻辑
- 某个模块专属字段解释
- 某个业务域专属转换逻辑

### 4.2 `auth`

职责：

- 登录
- 登出
- 当前用户
- 会话鉴权

不承担：

- 客户业务
- 跟进业务
- 看板聚合业务

### 4.3 `customer`

职责：

- 客户主数据
- 客户标签
- 客户详情
- 客户负责人相关信息
- 客户下次跟进时间等基础客户属性

不承担：

- 看板聚合展示
- 公共安排
- 报表提交

### 4.4 `followup`

职责：

- 跟进记录新增
- 跟进记录列表/时间线
- 跟进动作带来的客户跟进字段联动

允许与 `customer` 协作，但只能通过客户域 `service` 进行联动。

### 4.5 `task`

职责：

- 公共安排
- 任务看板
- 状态流转

不承担：

- 首页看板聚合
- 日报周报

### 4.6 `report`

职责：

- 日报
- 周报

不承担：

- 任务看板
- 话术库

### 4.7 `script`

职责：

- 话术分类
- 话术内容

不承担：

- 客户详情
- 看板统计

### 4.8 `dashboard`

职责：

- 首页聚合查询
- 首页统计口径
- 首页提醒列表

`dashboard` 是聚合层，不是基础业务层。  
它可以读 `customer / followup / task`，但不能反过来让这些业务域依赖 `dashboard`。

## 5. 标准分层职责

每个业务模块统一使用以下分层：

```text
module/
├── controller/
├── service/
├── service/impl/
├── dto/
├── vo/
├── entity/
├── mapper/
└── convert/
```

职责约束如下。

### 5.1 `controller`

只负责：

- 路由接入
- 请求参数接收
- 调用 `service`
- 返回统一响应

不允许：

- 写业务规则
- 直接操作 `mapper`
- 手写跨模块拼装逻辑

### 5.2 `service`

只负责：

- 业务规则
- 模块内流程编排
- 跨模块调用协调

不允许：

- 依赖别的模块 `impl`
- 直接暴露数据库细节给 controller

### 5.3 `dto`

只负责：

- 请求参数
- 查询条件
- 创建/更新入参

### 5.4 `vo`

只负责：

- 对外返回结构
- 页面展示所需响应字段

规则：

- `vo` 主要面向接口返回
- 不把 `vo` 当成长期内部领域模型滥用
- `P0` 阶段允许 `service` 临时直接返回 `vo`
- 但后续真实持久化阶段应尽量通过 `entity + convert` 收口

### 5.5 `entity`

只负责：

- 数据库实体
- 表字段映射

必须优先继承：

- `com.yousells.common.entity.BaseEntity`

### 5.6 `mapper`

只负责：

- 持久化读写
- SQL 映射

不允许：

- 承载业务规则

### 5.7 `convert`

只负责：

- `entity / dto / vo` 之间转换

不允许：

- 承载流程控制逻辑

## 6. 跨模块调用规范

### 6.1 允许的依赖方向

允许：

- `controller -> service`
- `service impl -> 本模块 mapper`
- `service impl -> 对方模块 service interface`
- `dashboard service -> customer/followup/task service interface`

### 6.2 不允许的依赖方向

不允许：

- `controller -> mapper`
- `controller -> 其他模块 service impl`
- `service impl -> 其他模块 controller`
- `service impl -> 其他模块 mapper`
- `customer/followup/task/report/script -> dashboard`

### 6.3 聚合逻辑规则

以下逻辑统一收口到 `dashboard/**`：

- 首页统计
- 首页客户提醒排序
- 首页任务提醒排序

业务模块只提供自己的真实数据，不主动写首页逻辑。

---

## 7. P0 阶段特别约束

### 7.1 允许的临时方案

在 `P0` 阶段允许存在：

- 部分 `service` 先用样例数据过渡
- `dashboard` 通过现有 service 查询结果做聚合
- 某些页面接口先返回满足联调的最小字段集

### 7.2 不允许继续扩散的问题

从现在开始不允许继续新增以下坏味道：

- 新业务逻辑继续堆在 `controller`
- 新模块继续只写 `vo` 不落 `entity/mapper`
- 为了方便联调直接跨模块改别人目录
- 前端缺字段时后端成员跑去改 `dashboard/**` 或共享入口文件
- 看板统计写回业务模块中

### 7.3 必须优先补齐的标准化点

后续真实化时优先补：

- `entity`
- `mapper`
- `convert`
- 模块测试
- 明确字段口径

## 8. 文件所有权与写入边界

### 8.1 队长 `秦梓源`

独占：

- `yousells-server/src/main/java/com/yousells/common/**`
- `yousells-server/src/main/java/com/yousells/modules/dashboard/**`
- `yousells-server/src/test/java/com/yousells/modules/dashboard/**`
- `docs/**`
- `docker-compose.yml`
- 部署与环境基线文件

职责：

- 基础设施
- 公共规范
- 看板聚合
- 最终联调与验收

### 8.2 志明

独占：

- `yousells-server/src/main/java/com/yousells/modules/customer/**`
- `yousells-server/src/main/java/com/yousells/modules/followup/**`

职责：

- 客户域
- 跟进域

### 8.3 哲涛

独占：

- `yousells-server/src/main/java/com/yousells/modules/task/**`
- `yousells-server/src/main/java/com/yousells/modules/report/**`
- `yousells-server/src/main/java/com/yousells/modules/script/**`

- `yousells-server/src/test/java/com/yousells/modules/script/**`

- `yousells-server/src/main/resources/mapper/report/**`
- `yousells-server/src/main/resources/mapper/script/**`

- 协作域

- 话术域

### 8.4 边界处理规则

如果必须改到别人所有权文件：

1. 先提给队长
2. 由队长判断是补共享基线，还是拆新文件
3. 不允许开发者自己跨目录顺手改

## 9. 后端任务下发前的统一验收口径

队员开始开发前，必须全部接受以下规则：

- 只在自己的独占目录内写代码
- 新模块先补 `entity / mapper / convert`
- 写完模块立刻补自动化测试
- 如果接口字段变化，先同步文档再联调
- 页面相关接口改动必须提醒前端做联调

队长验收后端模块时，至少检查：

- 是否越界改了别人目录
- 是否把业务逻辑写进 controller
- 是否把聚合逻辑写进业务域
- 是否补了测试
- 是否改了字段却没同步文档

## 10. 对当前项目的直接结论

当前 `YouSells` 后端：

- 已经具备继续下发任务的基础
- 不需要停下来做大重构
- 但必须按本规范执行后续开发

也就是说：

- 现在可以分发任务
- 但分发时必须连这份边界规范一起发
- 后面谁越界改共享文件、跨模块乱写、跳过测试，都按不合格处理

## 11. 下一步建议

正式下发任务时，建议你同时发 3 份东西：

1. `docs/11_后端架构收口与协作边界规范.md`
2. `docs/06_P0开发任务分配方案.md`
3. `docs/05_开发流程与日志规范.md`

这样队员拿到的不是“去做某个功能”，而是：

- 做什么
- 在哪里做
- 不能碰哪里
- 做完怎么验收

这才是你这一轮最重要的架构收口成果。
