# 鉴权与权限模型

## 1. 当前鉴权策略

当前明确策略：

- `v1`：`Spring Security + Session/Cookie`
- `后续预留`：`JWT + Redis + 网关 / 认证中心`

这意味着第一版先用最稳、最好维护的方案上线，但在代码结构上预留未来升级空间。

## 2. 为什么 v1 不直接上 JWT

因为当前场景是：

- 内部管理平台
- 用户规模 `50-100`
- 主要是浏览器访问
- 更重视稳定上线和维护成本

在这个阶段，`Session/Cookie` 更适合：

- 会话失效控制
- 强制下线
- 权限即时生效
- 并发登录控制
- 维护和排错

## 3. 如何给 JWT 预留空间

### 3.1 不把业务代码写死在 Session 里

- 控制层和业务层不要到处直接取 `HttpSession`
- 统一通过认证上下文工具拿当前用户
- 权限判断统一走一套服务或注解

### 3.2 先把用户权限模型设计好

至少要有：

- 用户表
- 角色表
- 权限点表
- 用户角色关系表
- 角色权限关系表

### 3.3 接口层保持认证方式可替换

- 登录、登出、获取当前用户信息统一走 `/auth`
- 前端只关心“是否登录、当前是谁、拥有哪些权限”
- 不把前端逻辑写死成某一种认证方式

## 4. 当前建议的角色

第一版建议至少有两个角色：

- `ADMIN`
  队长 / 管理员
- `MEMBER`
  普通成员

后续如果需要再细分：

- `LEADER`
- `SALES`
- `OPS`

## 5. 当前建议的权限规则

### 所有人都可以

- 查看客户总表
- 查看公共安排
- 查看话术库
- 提交日报周报

### 客户负责人可以

- 编辑自己负责的客户核心信息
- 写跟进记录
- 修改下次跟进动作

### 协助人可以

- 补跟进记录
- 补反馈
- 看客户详情

### 管理员可以

- 改负责人
- 改标签
- 改阶段
- 看全部报表
- 看全部客户推进情况

## 6. 一客一负责人原则

第一版建议严格执行：

- 一个客户同一时刻只绑定一个主负责人
- 可以有协助人
- 转负责人必须留记录
- 长时间未跟进要提示
