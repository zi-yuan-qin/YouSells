# YouSells P0 开发任务分配与进度管理方案

基于 `1 名队长统筹基础设施与验收 + 2 名后端并行开发 + 2 名前端并行开发` 的协作模式  
日期：`2026 年 5 月 19 日`

---

## 一、P0 产品定位与开发总览

### 1.1 P0 目标

`YouSells` 的 `P0` 不是做完整平台，而是先把内部业务中台最核心的一条主流程打通：

`登录 -> 首页看板 -> 客户总表 -> 客户详情 -> 跟进记录 -> 公共安排 -> 日报/周报 -> 话术库`

P0 目标是：

- 把项目基础设施搭起来
- 把权限和数据口径先统一
- 把核心业务模块全部跑通
- 把自动化测试流程强制带起来
- 为后续 `P1 / P2` 扩展留稳定基础

### 1.2 P0 范围内功能

模块 | 功能 | 独立性
--- | --- | ---
模块 A | 登录与权限基础 | 共享基础
模块 B | 首页看板 | 依赖共享接口
模块 C | 客户总表 + 客户详情 | 核心业务
模块 D | 跟进记录 | 依赖客户模块
模块 E | 公共安排区 | 相对独立
模块 F | 日报 / 周报 | 相对独立
模块 G | 话术库 | 相对独立
公共模块 | 工程基线、数据库设计、接口规范、测试规范、GitHub 协作、部署基线 | 共享

### 1.3 P0 不做功能

以下内容不属于 `P0` 范围：

- 月报
- 复盘总结区
- 匿名意见箱
- 工作聊天室
- 知识分享区
- 博客 / 日常分享区
- 微服务拆分
- refresh token / Redis 黑名单 / 多端登录踢出策略
- 对外开放 API

- `许润`：客户详情 / 跟进 / 公共安排 / 日报周报 / 话术库相关页面、组件、API、类型文件

### 1.5 当前协作原则

- `秦梓源` 负责每个阶段的基础设施准备和最终测试验收
- 模块开发者完成代码后，必须立刻写模块自动化测试
- 所有页面相关改动，必须明确提醒开发者做前后端联调
- 开发前先定需求、开发规划、开发日志，再写业务代码
- 每个模块都必须有负责人、依赖项、验收标准、状态管理
- 新增目录与文件必须按本方案执行，避免每个人自己起风格
- 本轮任务分配以“目录独占 + 文件独占”为第一原则，不按“大家一起改 customer 域”这种方式分工
- 共享文件默认归队长统一维护，前端成员不要在自己的 PR 中顺手改 `router / layout / app-shell / request / 共享类型`
- 任何人如果发现必须修改他人所有权文件，先提给队长，由队长决定是先补共享基线还是拆出新文件
- 一个 PR 只解决一个任务包，不做跨模块顺手重构

分发任务前必须统一阅读：

- `docs/11_后端架构收口与协作边界规范.md`
- `docs/05_开发流程与日志规范.md`
- `docs/08_API接口基线.md`

## 十一、当前阶段结论

当前 `YouSells` 的 `P0` 开发，不是简单按页面拆活，而是按 `基础设施 -> 目录与文件所有权切开 -> 后端真实化 -> 前端页面独占开发 -> 队长统一挂路由与聚合 -> 联调 -> 测试验收` 这条完整链路推进。

如果你现在要直接派任务，可以直接按下面这组说法分：

1. `秦梓源`
   不动现有基础设施，继续负责 `dashboard/**` 聚合、前端共享壳子、路由挂载、最终联调与验收
2. `志明`
   只做 `customer/** + followup/**`，从实体、Mapper、Convert 到接口真实化一条线做完
3. `哲涛`
   只做 `task/** + report/** + script/**`，不碰客户域和 dashboard 聚合
4. `嘉诚`
   只做登录、看板、客户总表，使用独立的 `customer-list` 组件、API、类型文件
5. `许润`
   只做客户详情、跟进、任务、日报、话术库，使用独立的 `customer-detail` 组件、API、类型文件

这版文档的目的，不只是让每个人知道自己要做什么，更重要的是让每个人知道“哪些文件不能碰”，从源头减少 PR 合并冲突。
